# Mobile App Development Process

**A simple, mostly-automated workflow for a two-person team.**
Managed with Claude, integrated with GitHub, Replit, and Notion.

> Goal: ship small, well-built, customisable mobile apps (iOS + Android) with the least possible overhead. Two people, four phases, clear hand-offs.

---

## The Team (2 people, many hats)

| Person | Primary Role | Also Plays |
|--------|--------------|------------|
| **Denys** | **Designer** — creates mockups in Figma, exports screenshots + design specs | Business Analyst, Product Manager, QA (visual) |
| **Michel** | **Developer** — builds and deploys the app on iOS + Android | Business Analyst, Product Manager, QA (functional), Release Manager |

Both act as **Business Analyst / Product Manager** during Phase A and Phase B to refine and approve requirements. Both act as **QA** before release.

---

## The Mindsets (apply in every phase)

**Reuse**
- Component apps — build features as reusable component apps.
- Themes to adapt to new clients — same layout, swap colour tokens.
- Master database to scale data and enable data analysis.

**Agile but not agile**
- Learn good fundamentals — don't rush, learn them well.
- Create simple, customised solutions *with* agility (because they're simple).

**Simplicity & User-Centric**
- Customise based on real user interaction (analytics, AI).
- Reduce the user's margin of error — turn off unnecessary UI elements.

**Continuous Improvement**
- Track usage with analytics.
- Run an AI agent to track Reddit complaints and turn them into features.

**Experimentation**
- Failure is part of the business.
- Marketing is 95% of conversion success.

## The Four Phases

```
  A. USER STORIES  ──►  B. SPECS  ──►  C. ARCHITECTURE  ──►  D. PRODUCTION
   (who & why)         (what)          (how)                 (ship & improve)
   USER_STORIES.md     SPECS.md        ARCHITECTURE.md       update all 3
   Denys + Michel      Denys + Michel  Michel (Denys QA)     Both
   as BA/PM            as BA/PM        builds                ship + monitor
```

Each phase produces or updates a Markdown file. Each file feeds the next. The loop closes in Phase D, which sends learnings back to A, B, and C.

### Phase A — User Stories → `USER_STORIES.md`
Describe personas, user behaviour, and use cases as simply as possible. Both people act as BA/PM to refine and **approve** the stories. Includes a **Monetisation** section. *See the file template for the sections to fill.*

### Phase B — Specs → `SPECS.md`
A simpler version of a full build spec (screens, UX, theme tokens, data, core logic, roadmap). Fill the template, approve, hand to architecture.

### Phase C — Architecture → `ARCHITECTURE.md`
Use `USER_STORIES.md` + `SPECS.md` to design the system: navigation, state, data, backend, build/deploy. Produce mockups (Figma) and scaffolding files (often generated by Replit). Locks the tech stack.

### Phase D — Production & Continuous Improvement
Publish to the App Store and Google Play, monitor crashes/analytics/reviews, run the Reddit-complaint agent, and **update all previous `.md` files** as the app evolves.

## The Automated Workflow (Claude + GitHub + Replit + Notion)

The idea: Claude is the connective tissue. Humans decide and approve; Claude drafts, scaffolds, syncs, and watches.

| Tool | Role in the workflow |
|------|----------------------|
| **Claude** | Drafts and updates the `.md` files, scaffolds code, writes EAS/CI config, summarises reviews and crashes, runs the Reddit agent. The "operator" of the workflow. |
| **Figma** | Denys' design source of truth. Screenshots + specs exported here. |
| **Replit** | Fast prototyping and scaffolding. Generates the initial Expo project structure to import. |
| **GitHub** | Code + the `.md` docs live here. Single source of truth. PRs = approval gates. EAS builds trigger from here. |
| **Notion** | Human-facing dashboard: phase status, backlog, approvals, release notes, review/complaint digests. Mirrors the `.md` files for non-technical reading. |
| **EAS** | Build, submit, and OTA-update the app to both stores. |

### Step-by-step flow

1. **Kickoff (Phase A).** Claude opens `USER_STORIES.md` from the template. Denys + Michel fill personas/use cases/monetisation in a working session. Claude tidies it, commits to GitHub, mirrors a readable copy to Notion. **Approval gate:** both tick the approval box.

2. **Specs (Phase B).** Claude generates `SPECS.md` from the approved stories + the template. Denys attaches Figma screenshots and tokens. Claude commits + mirrors to Notion. **Approval gate.**

3. **Architecture (Phase C).** Claude reads both files and writes `ARCHITECTURE.md` (stack, data model, folder structure). Replit scaffolds the Expo project; Claude reconciles it with the architecture doc and pushes to GitHub. Denys QAs the mockups against the build.

4. **Build & Deploy (Phase C→D).** Michel + Claude implement screen by screen. EAS builds to TestFlight + Google Play internal testing. Small fixes ship via **EAS Update** (no re-submission); native changes go through a full EAS build + store submit.

5. **Production loop (Phase D).** Vexo (analytics), Sentry (crashes), and store reviews feed a weekly digest. Claude's Reddit agent scans for complaints. Findings become new user stories → the loop restarts at Phase A for the next version. Claude updates all three `.md` files and the Notion dashboard.

### Approval gates (keep it lightweight)
Every phase ends with a one-line approval in the file:

```
## Approval
- [ ] Denys (Design/BA)  — date:
- [ ] Michel (Dev/BA)    — date:
```

A phase is "done" only when both boxes are checked. This is the entire governance model — no extra ceremony.

## File map

```
Apps Development/
├── PROCESS.md          ← this file (the workflow itself)
├── USER_STORIES.md     ← Phase A template + example
├── SPECS.md            ← Phase B template + example
├── ARCHITECTURE.md     ← Phase C template + example
└── apps/
    └── <app-name>/     ← scaffolded Expo project (GitHub repo)
```

For each **new app**, copy `USER_STORIES.md`, `SPECS.md`, and `ARCHITECTURE.md` into `apps/<app-name>/docs/` and fill them in. The templates stay clean at the root for reuse (the "Reuse" mindset in practice).
